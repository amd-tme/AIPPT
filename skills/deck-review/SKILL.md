# Deck Review & Edit

This skill covers two related workflows:

1. **Visual Review** — inspect a deck's slides, content, and layout; identify issues; provide feedback
2. **Edit Deck** — modify slides by editing their generating code (slides-as-code) or directly via python-pptx

For slides-as-code decks this is the third step: `/create-deck` → `/deck-review` → `/edit-deck`.
For externally created decks (not generated by aippt), skip to [Visual Review](#visual-review).

---

## Visual Review

### Step R1: Locate Slide Images

**Always prefer PowerPoint-exported images over LibreOffice conversion.** LibreOffice introduces rendering artifacts (clipped text, wrong word-wrap, font substitution) that produce false positives in visual analysis.

Resolution order — use the first that succeeds:

#### 1. Pre-exported images from aippt ingest (best quality)

Check if slide images already exist from a prior `aippt ingest` or `aippt export-images` run:

```bash
# Images are stored at images/<deck-stem>/ where deck-stem is the filename without extension
# e.g. uploads/MyDeck.pptx → images/MyDeck/Slide1.PNG, Slide2.PNG, ...
ls images/<deck-stem>/
```

If `Slide1.PNG` (or `slide-1.png`) is present, use those images directly — they are native PowerPoint quality.

#### 2. Export images via aippt (requires PowerPoint on Windows)

If no pre-exported images exist, generate them:

```bash
$VENV_PYTHON aippt.py export-images <deck.pptx> images/<deck-stem>/
# Optional: higher resolution
$VENV_PYTHON aippt.py export-images <deck.pptx> images/<deck-stem>/ --width 2560 --height 1440
```

`aippt export-images` wraps PowerPoint's native image export via COM automation. **This only works on Windows with PowerPoint installed.** On Linux/WSL2, PowerPoint is not available — use option 3.

> **WSL2 tip:** If the PPTX lives in the WSL filesystem, you can run `aippt export-images` from a Windows terminal (PowerShell/cmd) pointing at a Windows-accessible path (e.g. `\\wsl$\Ubuntu\home\user\...`), then use the resulting images from WSL.

#### 3. LibreOffice fallback (Linux / no PowerPoint)

Only use this if PowerPoint-exported images are unavailable:

```bash
mkdir -p /tmp/deck-review
HOME=/tmp libreoffice --headless --convert-to pdf <deck.pptx> --outdir /tmp/deck-review
pdftoppm -jpeg -r 150 /tmp/deck-review/<deck>.pdf /tmp/deck-review/slide
```

⚠️ **LibreOffice rendering caveats — treat findings with extra skepticism:**
- Text clipping and word-wrap artifacts are common (especially with custom fonts)
- Font substitution changes text layout
- Some shapes and effects render differently or are missing
- Always re-verify any visual issue against PowerPoint-exported images before reporting it as a real bug

### Step R2: Extract Text Content

```bash
$VENV_PYTHON -m markitdown <deck.pptx>
```

This extracts all text content (titles, body, speaker notes) for content analysis independent of visual rendering.

### Step R3: Initial Visual Review

Read slide images in batches. For each slide note:
- **Layout issues:** text overflow, overlapping elements, empty placeholder text
- **Content issues:** filler/placeholder text left in, duplicate slides, missing content
- **Consistency issues:** mismatched templates, inconsistent slide numbering in footers, classification markers rendering as text (e.g. `{Public}`)
- **Design issues:** dense/unreadable layouts, poor visual hierarchy

### Step R3b: Speaker Notes Flow Review

After extracting text content (Step R2), read all speaker notes sequentially and evaluate:

- **Repetitiveness:** Flag notes that restate the same points from the previous slide's notes verbatim or near-verbatim. Each slide's notes should add new context, not echo what was just said.
- **Transition quality:** Notes should acknowledge where the narrative is moving — e.g. "Now that we've covered X, let's look at Y..." rather than abruptly starting a new topic.
- **Natural cadence:** Check that the length and tone of notes are roughly consistent. A 30-word note followed by a 300-word note is jarring unless intentional.
- **Opening/closing hooks:** The first slide's notes should orient the audience; the last slide's notes should land the key takeaway rather than trailing off.
- **Redundancy with slide content:** Notes that simply read the slide bullets aloud add no value — they should expand on, contextualize, or complement what's on the slide.

### Step R4: Report Findings

Present a structured summary:
- Overall assessment (structure, flow, branding consistency)
- Table of issues with: Slide #, Issue type (High/Medium/Low), Description
- **Notes flow summary:** A brief narrative assessment of the speaker notes as a whole — highlight any repetition patterns, abrupt transitions, or pacing problems
- Distinguish confirmed issues (visible in PowerPoint images) from suspected issues (LibreOffice only)
- Ask which slides the user wants to work on

## Environment Setup

```bash
# Detect platform and venv
if [ -f venv/bin/python ]; then
    VENV_PYTHON=venv/bin/python
elif [ -f venv/Scripts/python.exe ]; then
    VENV_PYTHON=venv/Scripts/python.exe
fi
```

## Invocation

Parse `$ARGUMENTS` for two parts: **target** and **edit request**.

```
/edit-deck output/deck.mjs — make slide 3 a numbered list
/edit-deck "Q1 Security Review" — add a summary slide at the end
/edit-deck output/deck.mjs — add speaker notes to every slide
/edit-deck output/deck.mjs — move compliance section before security
```

**Parsing rules:**
- If `$ARGUMENTS` contains ` — ` (space-emdash-space) or ` - ` (space-hyphen-space), split on the first occurrence: left = target, right = edit request
- If no separator, treat the entire argument as the target and ask for the edit request interactively
- If no arguments at all, ask for both target and edit request

## Workflow

### Step 1: Resolve Source

Use the `source_resolver` module to find the script:

```python
from aippt.source_resolver import resolve_source

result = resolve_source(target, db_path="slides.db")
```

**Handle each case:**

| Result | Action |
|--------|--------|
| `result["script_path"]` is set | Proceed to Step 2 |
| `result["error"]` with `"Multiple decks"` | Show `result["choices"]` and ask user to pick one (by ID) |
| `result["error"]` with `"No deck found"` | Ask user: "No deck found matching '{target}'. Provide a direct script path?" |
| `result["error"]` with `"No source script"` | Tell user: "This deck doesn't have source tracking. Provide the script path directly, or re-ingest with `--source`." |

### Step 2: Read Context

Read the source script file. This is the primary context for the edit.

**Always read:**
- The source script (the code being edited)

**Optionally read (based on the edit request):**
- **PPTX content** via `python -m markitdown <path>.pptx` — when the user references current slide content or when you need to see the rendered state
- **Slide images** — when the edit is about visual issues (text overflow, alignment, spacing). Use the image resolution priority from [Step R1](#step-r1-locate-slide-images): pre-exported PowerPoint images in `images/<deck-stem>/` first, then `aippt export-images`, then LibreOffice as a last resort (with caveats). **Never report a visual bug based solely on LibreOffice output** — verify against PowerPoint-exported images.
- **Theme YAML** at `themes/<name>.yaml` — when the edit involves colors, fonts, or styling
- **Original outline** (if `outline_path` is set in catalog) — when the edit is about content accuracy or intent

### Step 3: Plan the Edit

Identify what needs to change — do NOT modify the file yet:

1. Find the target slide(s) using the `// ═══ Slide N: Title ═══` markers
2. Determine the specific code changes needed
3. For pptxgenjs: follow the patterns in @references/edit-patterns.md
4. For python-pptx: follow standard python-pptx API patterns

**Important:** Only modify the relevant slide blocks. This keeps diffs clean and preserves the user's existing customizations.

### Step 4: Preview and Confirm

Describe the planned changes to the user: which slides will be touched, what will change in each.

Ask: "These changes look right? Apply and regenerate?"

### Step 5: Backup, Apply, and Diff

**First:** Create a backup before writing any changes:

```python
from aippt.source_resolver import create_backup
create_backup(script_path)
```

The backup is created only once (first edit in a session). If a `.bak` already exists, it's preserved — it represents the pre-edit baseline.

**Then:** Apply the edits using the Edit or Write tool.

**Finally:** Show what changed:

```bash
git diff <script-path>
```

Or if the file isn't tracked, summarize what was modified.

### Step 6: Regenerate

Run the script to produce the updated PPTX:

```python
from aippt.source_resolver import run_script
result = run_script(script_path, engine=engine)
```

**If successful:** Proceed to Step 7.

**If failed:** See @references/error-recovery.md for the full error handling flow:
1. Show the error output
2. Offer to restore from `.bak` backup
3. Offer to attempt an LLM fix of the syntax error
4. If file lock detected: ask user to close PowerPoint and retry

### Step 7: Update Metadata

After a successful edit + regeneration, update the speaker notes metadata:

```python
from pptx import Presentation
from aippt.notes_meta import parse_notes_meta, serialize_notes_meta, append_history

prs = Presentation(pptx_path)
for slide in prs.slides:
    if slide.has_notes_slide:
        notes_text = slide.notes_slide.notes_text_frame.text
        text, meta = parse_notes_meta(notes_text)
        if meta:
            append_history(meta, "<description of change>", "/edit-deck")
            slide.notes_slide.notes_text_frame.text = serialize_notes_meta(text, meta)
prs.save(pptx_path)
```

**Only update metadata for slides that were actually edited** — don't touch unchanged slides.

The change description should be concise and descriptive: "Changed to two_column layout", "Added speaker notes", "Rewrote bullets for exec audience", "Fixed text overflow".

### Step 8: Continue or Hand Off

After a successful edit, offer the user options:

```
Edit applied and deck regenerated: output/{name}.pptx

Options:
- Make another edit (describe what to change)
- /deck-review — Visual QA to verify the changes
- Done — commit the script + PPTX together
```

If the user provides another edit request, loop back to Step 3 (same script, same session). The `.bak` backup from the first edit is preserved across the session.

## Capabilities

### Single-Slide Edits
- Change layout type (bullet → two_column, etc.)
- Rewrite content / bullets
- Add/edit speaker notes
- Improve notes flow (reduce repetition, improve transitions between slides)
- Add/swap images
- Adjust styling (colors, fonts, spacing)
- Fix visual issues flagged by /deck-review

### Deck-Level Edits
- Add / remove / duplicate slides
- Reorder slides or sections
- Change theme or global styling
- Add section dividers
- Batch operations ("add notes to all slides")
- Insert new slides between existing ones

### Identifying Slides

Use the `// ═══ Slide N: Title ═══` comment markers to find slides in the script. Examples:

```
"slide 4" → find // ═══ Slide 4:
"the architecture slide" → find // ═══ Slide N: Architecture
"all slides" → iterate through all marker blocks
```

## Safety

- **Backup:** `.bak` created before first edit, preserved until user deletes it
- **Diff preview:** Always show changes before applying
- **Git-friendly:** Edited scripts produce clean diffs
- **File locks:** Detect PowerPoint file locks and prompt to close
- **Minimal edits:** Only modify the slide blocks that need changing

## Troubleshooting

| Problem | Cause | Fix |
|---------|-------|-----|
| Visual issue only visible in LibreOffice output | LibreOffice rendering artifact | Verify against PowerPoint-exported images in `images/<deck-stem>/` before reporting |
| `aippt export-images` fails on Linux/WSL2 | Requires PowerPoint on Windows | Use pre-ingested images in `images/` dir, or run export-images from Windows side via `\\wsl$\...` path |
| No images in `images/<deck-stem>/` | Deck not yet ingested or images not exported | Run `aippt export-images <deck.pptx>` on Windows, or `aippt ingest <deck.pptx>` with `--images-dir` |
| "No source script tracked" | Deck was ingested without `--source` | Re-ingest: `aippt ingest deck.pptx --source output/deck.mjs` |
| Script fails after edit | Syntax error introduced | Offer restore from `.bak` or LLM fix attempt |
| PPTX won't overwrite | File open in PowerPoint | Close PowerPoint, then retry |
| Wrong deck resolved | Ambiguous name match | Use deck ID instead of name |
| Script path in catalog but file missing | File was moved or deleted | See @references/error-recovery.md — offer to search output/ |
| Metadata not updating | No `---aippt-meta---` block in notes | Only present in decks created after PRD 1; older decks skip metadata |
| Node not found | Node.js not installed | Install Node.js or use python-pptx engine |
